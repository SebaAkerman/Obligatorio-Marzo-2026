1. Proyecto LOST — Aprendizaje por Refuerzo (Q-Learning)

1.1 Descripción del ambiente

El proyecto LOST utiliza el ambiente MountainCarContinuous-v0 de Gymnasium. Un auto (nuestro rover) se encuentra en un valle entre dos colinas y debe llegar a la bandera ubicada en la cima de la colina derecha. El motor del auto no es lo suficientemente fuerte para subir directamente, por lo que el agente debe aprender a generar impulso, retrocediendo y avanzando alternadamente.

ElementoDescripción
Espacio de observaciónBox(2,): posición x ∈ [-1.2, 0.6], velocidad v ∈ [-0.07, 0.07]
Espacio de accionesBox(1,): fuerza continua a ∈ [-1.0, 1.0]
Recompensa por pasor = -0.1·a² (penaliza el uso de fuerza)
Recompensa por meta+100 al alcanzar x ≥ 0.45
Fin de episodioterminated=True (éxito, llegó a la meta) o truncated=True (máx. 999 pasos)

El desafío central de este ambiente es que la recompensa es esparsa y mayormente negativa: mientras el agente no alcance la meta, todo lo que recibe son penalizaciones por usar fuerza. Esto puede inducir a un agente mal inicializado a aprender que "no moverse" (a ≈ 0) es la política óptima, quedando atrapado en un mínimo local — problema que se analiza en profundidad en la Sección 4.

1.2 Discretización de observaciones y acciones

Dado que Q-Learning tabular requiere un espacio de estados y acciones discreto y finito, se implementó la clase Discretizer (utils/discretization.py), que:

  • Genera grillas uniformes con np.linspace para posición (n_pos_bins) y velocidad (n_vel_bins) dentro de sus rangos válidos.
  • Convierte una observación continua (x, v) en un índice de estado discreto (x_bin, vel_bin) mediante np.digitize.
  • Genera n_actions valores de fuerza uniformemente distribuidos en [-1, 1], y mapea el índice de acción discreto elegido por la política a la acción continua que espera env.step().

Se exploraron seis configuraciones de discretización, entrenando 5.000 episodios con los mismos hiperparámetros de aprendizaje (α=0.2, γ=0.99, ε₀=1.0, decay=0.9995, ε_min=0.05, q_init=1.0):

El espacio de estados crece como O(bins²×acciones), pero la representación útil crece más lento porque muchos estados nunca se visitan. Bins de 10×10 producen aliasing (estados físicamente distintos colapsan al mismo bin), perdiendo la resolución necesaria cerca de la meta. Acciones=5 discretiza la fuerza de forma demasiado gruesa. Elección final: 30×30 bins, 15 acciones — mejor reward con un espacio de estados todavía manejable; 20 acciones agrega exploración redundante sin mejora apreciable.

1.3 Q-Learning

Se implementó QLearningAgent (q_learning_agent.py), un agente Q-Learning tabular con tabla Q de forma (n_pos_bins+1, n_vel_bins+1, n_actions) y política ε-greedy. La regla de actualización (Sutton & Barto, ec. 6.8) es:

Q(s, a) ← Q(s, a) + α · [ r + γ · maxa' Q(s', a') − Q(s, a) ]

La exploración se controla con ε-greedy: con probabilidad ε se elige una acción aleatoria, y con probabilidad 1-ε la acción de mayor valor Q. ε decae multiplicativamente cada episodio (epsilon ← max(epsilon_min, epsilon · decay)), permitiendo que el agente explore mucho al inicio y explote su política aprendida hacia el final.

Un parámetro adicional, q_init, controla el valor inicial de toda la tabla Q. Como se detalla en la Sección 4, la inicialización optimista (q_init=1.0) resultó ser crítica para que el agente no quede atrapado en la política trivial "fuerza = 0".

1.5 Componente de investigación: Dyna-Q

Siguiendo los capítulos 8.1 y 8.2 de Reinforcement Learning: An Introduction (Sutton & Barto), se implementó DynaQAgent (dyna_q_agent.py), que extiende Q-Learning integrando actuación, aprendizaje y planificación. Por cada paso real (s, a, r, s'):

  1. Actualización real: se aplica la misma regla de Bellman que Q-Learning sobre la transición observada.
  2. Actualización del modelo: se guarda Model(s,a) ← (r, s') (modelo determinista, tabular).
  3. Planificación simulada: se repiten n veces los siguientes pasos:
    • Samplear aleatoriamente un par (s̃, ã) visitado anteriormente.
    • Recuperar (r̃, s̃') del modelo.
    • Aplicar la actualización de Q sobre esta experiencia simulada.

Con n=0, Dyna-Q se reduce exactamente a Q-Learning puro. El parámetro n_planning_steps controla cuánta "experiencia simulada" extra se reutiliza por cada paso real, acelerando la propagación de información hacia atrás sin necesidad de más interacción con el ambiente.

Sweep de n_planning_steps

Hallazgo central: más planificación no es siempre mejor. Con n<10 el modelo aprendido es demasiado pobre y la planificación no alcanza a propagar el valor de la meta. Con n=10 se logra el balance óptimo (66% de éxito a 3.000 episodios, 100% al extender a 5.000). El n óptimo depende del presupuesto de episodios reales y de cuán denso esté el modelo aprendido en ese punto del entrenamiento.

2. Proyecto MATE — Minimax / Expectimax (Isolation)

2.1 Descripción del ambiente

El proyecto MATE utiliza un simulador del juego de mesa Isolation sobre un tablero de 4×4 (Board en board.py, expuesto vía IsolationEnv). Dos jugadores ocupan celdas distintas y se mueven alternadamente. En cada turno, un jugador:

  1. Se mueve a una celda adyacente libre (8 direcciones posibles), y
  2. Destruye (inutiliza) una celda vacía cualquiera del tablero.

Una acción es entonces el par (dirección, celda_a_destruir). Un jugador pierde cuando, en su turno, no tiene ningún movimiento válido disponible. El factor de ramificación es alto para un tablero tan pequeño: hasta ~96 acciones posibles por nodo (8 direcciones × hasta 12 celdas vacías destruibles), lo que hace que el árbol de búsqueda crezca muy rápido con la profundidad.

2.2 Minimax con Alpha-Beta Pruning

Se implementó MinimaxAgent (minimax_agent.py), que asume que ambos jugadores juegan de forma óptima:

  • Nodos Max (nuestro agente): eligen la acción que maximiza la utilidad.
  • Nodos Min (el oponente): eligen la acción que minimiza la utilidad desde la perspectiva de nuestro agente.
  • Si el nodo es terminal, la utilidad es +∞ si ganamos o -∞ si perdemos. Si se alcanza la profundidad máxima sin terminar, se evalúa el estado con una función heurística (Sección 2.4).

Alpha-Beta Pruning mantiene dos cotas durante el recorrido: α (la mejor garantía que Max puede asegurarse hasta el momento) y β (la mejor garantía de Min). Una rama se poda cuando:

  • En un nodo Max, el valor parcial ≥ β → el padre Min nunca elegiría llegar aquí (poda β).
  • En un nodo Min, el valor parcial ≤ α → el padre Max nunca elegiría llegar aquí (poda α).

Esto reduce la complejidad de O(bd) a, en el mejor caso, O(bd/2) sin afectar el resultado: Alpha-Beta siempre devuelve la misma acción que Minimax puro, solo evita explorar ramas irrelevantes. Adicionalmente se implementó Move Ordering opcional (use_move_ordering=True): antes de expandir, las acciones se ordenan según su valor heurístico, de forma que las mejores jugadas se evalúen primero — esto ajusta α/β más rápido y, en teoría, poda más ramas.

Conclusiones:

  • A profundidad 4, Minimax puro tarda 45.6 segundos por partida — completamente inviable en tiempo real (14× más lento que profundidad 3). Alpha-Beta es obligatorio para depth ≥ 4: reduce los nodos un 97% y el tiempo a 1.15 s/partida, un agente jugable.
  • La paradoja del Move Ordering: aunque reduce un 99.7% los nodos (vs. 97% de AB solo), el tiempo de pared es mayor (1.94 s vs. 1.15 s). El costo de evaluar la heurística para ordenar cada acción (una llamada extra a h_mobility por acción, en cada nodo) supera el beneficio de la poda adicional cuando la heurística es tan barata como h_mobility (O(1)). Move Ordering sería más rentable con heurísticas costosas como h_territory o h_future_mobility.
  • El agente final (Sección 2.6) usa Alpha-Beta sin Move Ordering, por ser la configuración más eficiente con la heurística elegida.

2.3 Expectimax

Se implementó ExpectimaxAgent (expectimax_agent.py), que difiere de Minimax en los nodos del oponente: en lugar de un nodo Min (oponente óptimo), se usa un nodo de azar que calcula la esperanza de los valores de los hijos, asumiendo que el oponente elige sus acciones con distribución uniforme:

Expectimax(s) =
  U(s)                     si s es terminal
  maxa Expectimax(suc(s,a))    si jugador(s) = Max
  Σa P(a)·Expectimax(suc(s,a))   si jugador(s) = Azar (oponente), con P(a) = 1/|acciones|

Cuándo conviene cada técnica: Minimax es conservador — asume el peor caso (oponente óptimo). Expectimax es más optimista — adecuado cuando el oponente es aleatorio o subóptimo. Para decidir cuál usar en MATE se realizó un experimento riguroso (100 partidas balanceadas: 50 como P1 + 50 como P2, test binomial):

Conclusión: dado que el oponente real es otro agente que busca jugar bien (otro Minimax o Stratagem), la suposición de Expectimax (oponente uniforme/aleatorio) es incorrecta en este dominio. Minimax con Alpha-Beta es la técnica elegida para el agente final. Expectimax solo sería preferible contra oponentes genuinamente aleatorios o subóptimos.

2.6 Conclusiones — Proyecto MATE

Agente final elegido:

MinimaxAgent(depth=4, heuristic=eval_mobility_only, use_alpha_beta=True, use_move_ordering=False)
  • Alpha-Beta es esencial: sin él, depth=4 toma 45.6 s/partida (inviable); con AB, 1.15 s/partida (-97% de nodos).
  • Las heurísticas son estadísticamente equivalentes en 4×4 a depth=3 (Bonferroni, p>0.005 en los 10 pares) → se elige mobility_only por simplicidad y velocidad, sin penalización empírica.
  • La profundidad es el factor decisivo, más que la heurística: depth=4 supera a depth=3 con p=0.0007.
  • Minimax >> Expectimax a depth=3 (94% de victorias, p≈0.0000): el oponente real juega casi óptimo, por lo que la suposición de Expectimax (oponente aleatorio) es incorrecta en este dominio.
  • El agente final venció a Stratagem (agente de referencia de la cátedra) en el 76% de 100 partidas balanceadas (p≈0), y a RandomAgent en el 85%.
  • Ventaja estructural del primer jugador: ~73% en tablero 4×4 a depth=4 — por eso todos los torneos se balancean 50% P1 / 50% P2.

4. Advertencias y dificultades encontradas

Proyecto LOST

1. q_init=0 nunca converge Resuelta

Q-Learning con la tabla Q inicializada en 0 no llega a la meta en ninguna configuración probada. Como r = -0.1·a² es negativo para cualquier acción con fuerza, la primera vez que se visita un par (s,a) con |a|>0, su valor Q se vuelve negativo. Con ε decayendo, el agente termina prefiriendo estados aún no visitados (Q=0, "neutro") sobre acciones con fuerza, aprendiendo que fuerza ≈ 0 minimiza la penalización — un mínimo local del que nunca escapa, porque el reward +100 de la meta jamás llega a propagarse.

Solución: q_init=1.0 (inicialización optimista). Como 1.0 > cualquier valor negativo ya observado, el agente se ve "obligado" a seguir explorando cada estado antes de concluir que es malo, hasta que eventualmente alcanza la meta y el valor +100 comienza a propagarse hacia atrás.

2. γ < 0.99 hace la meta invisible Resuelta

Con γ=0.90, el valor descontado de la recompensa de la meta a ~500 pasos de distancia es 0.9⁵⁰⁰ ≈ 10⁻²³ — numéricamente indistinguible de cero en los TD-updates. Solo con γ≥0.99 (0.99⁵⁰⁰ ≈ 0.0066) la señal de la meta logra "verse" desde estados lejanos. Este es el hiperparámetro más sensible del proyecto: con γ∈{0.90, 0.95}, la tasa de éxito es 0% sin importar cuántos episodios se entrenen.

3. Dyna-Q: sensibilidad al estado del generador aleatorio Descubierta en validación

Al validar la robustez con múltiples semillas (random.seed + np.random.seed), Dyna-Q con q_init=0 falló consistentemente: 0% de éxito en las 3 semillas probadas con 5.000 episodios. El modelo dynaq_FINAL.pkl entregado fue entrenado sin semilla fija y convergió gracias a un estado del RNG favorable. La causa raíz: Dyna-Q necesita encontrar la meta al menos una vez mediante exploración aleatoria para que la planificación tenga algo que propagar; con un RNG desfavorable, esa primera llegada nunca ocurre dentro del presupuesto de episodios y el modelo de planificación queda "vacío" de información útil.

Solución verificada: con q_init=1.0, Dyna-Q se vuelve robusto y reproducible entre semillas (ver tabla multi-semilla a continuación).

4. Dyna-Q: curva no monótona de n_planning Resuelta

Como se muestra en la Sección 1.5, n_planning=20 produce peor desempeño (18% éxito) que n_planning=10 (66% éxito) bajo el mismo presupuesto de episodios. Con un modelo todavía poco denso, demasiada planificación refuerza prematuramente valores Q basados en pocas transiciones observadas, generando una política subóptima de la que es difícil escapar luego. n=10 resultó ser el punto óptimo para 3.000-5.000 episodios.

Proyecto MATE

5. La paradoja del Move Ordering Resuelta

Move Ordering reduce un 99.7% los nodos expandidos respecto a Minimax puro (vs. 97% de AB solo a depth=4), pero resulta más lento en tiempo de pared (1.94 s vs. 1.15 s/partida). El overhead de calcular la heurística para ordenar las acciones en cada nodo supera el beneficio de la poda adicional cuando la heurística es O(1) (como h_mobility). Conclusión: Move Ordering solo se justifica con heurísticas costosas (h_territory, h_future_mobility); con la heurística elegida (mobility_only), Alpha-Beta solo es la mejor configuración.

6. Ventaja estructural del primer jugador No resuelta — mitigada

En el mirror match (mobility_only depth=4 vs. sí mismo), P1 gana el 73% de las partidas (p≈0). Esto es inherente al tablero 4×4: el primer jugador siempre tiene acceso a más "territorio virgen". No es un problema de los agentes implementados, sino una propiedad del entorno. Se mitiga balanceando todos los torneos (50% de las partidas con cada agente como P1), de modo que las comparaciones entre heurísticas/profundidades no estén sesgadas por este efecto.

7. Las heurísticas son estadísticamente equivalentes Hallazgo, no dificultad

El experimento riguroso (Sección 2.4) mostró que las 5 heurísticas implementadas son equivalentes en 4×4 a depth=3 (ningún par supera α=0.005 post-Bonferroni). Esto fue inesperado — la hipótesis inicial era que eval_full o eval_territory serían claramente superiores por incorporar más señales. Con Alpha-Beta a mayor profundidad, el lookahead compensa la calidad de la evaluación de las hojas, haciendo que la elección de heurística sea, en la práctica, secundaria frente a la profundidad de búsqueda.

8. depth=5 mejora sobre depth=4 en 1v1, pero no contra Stratagem Hallazgo de extensión

Como extensión, se evaluó mobility_only depth=5: vence a depth=4 en enfrentamiento directo (62%, p=0.016), pero contra Stratagem obtiene 74% — estadísticamente indistinguible del 76% de depth=4 (intervalos de confianza solapados). El costo computacional adicional de depth=5 no se traduce en una mejora medible contra el benchmark externo, por lo que depth=4 se mantiene como el punto óptimo práctico elegido para el agente final. Datos en models/mate/depth5_results.csv.

def obs_to_state(self, obs: np.ndarray) -> tuple[int, int]:
    """Convierte una observación continua en un índice de estado discreto."""
    x, vel = obs
    x_bin = int(np.digitize(x, self.pos_bins))
    vel_bin = int(np.digitize(vel, self.vel_bins))
    return x_bin, vel_bin

def action_index_to_continuous(self, idx: int) -> np.ndarray:
    """Convierte un índice de acción discreta en la acción continua del ambiente."""
    return np.array([self.actions[idx]])

Regla de actualización Q-Learning (q_learning_agent.py)

def _update(self, obs, action_idx, reward, next_obs, done):
    state = self.disc.obs_to_state(obs)
    next_state = self.disc.obs_to_state(next_obs)

    max_next_q = 0.0 if done else float(np.max(self.Q[next_state]))
    target = reward + self.gamma * max_next_q
    self.Q[state][action_idx] += self.alpha * (target - self.Q[state][action_idx])

Paso Dyna-Q: actualización real + modelo + planificación (dyna_q_agent.py)

def _dyna_step(self, obs, action_idx, reward, next_obs, done):
    state = self.disc.obs_to_state(obs)
    next_state = self.disc.obs_to_state(next_obs)

    # 1. Actualizacion Q con experiencia real
    self._q_update(state, action_idx, reward, next_state, done)

    # 2. Actualizar modelo
    self.model[(state, action_idx)] = (reward, next_state, done)
    if (state, action_idx) not in self._visited_set:
        self._visited_set.add((state, action_idx))
        self._visited.append((state, action_idx))

    # 3. n pasos de planificacion simulada
    for _ in range(self.n_planning):
        if not self._visited:
            break
        s, a = random.choice(self._visited)
        r_sim, s_next_sim, done_sim = self.model[(s, a)]
        self._q_update(s, a, r_sim, s_next_sim, done_sim)

Minimax con Alpha-Beta Pruning (minimax_agent.py)

def _max_value(self, board, depth, alpha, beta):
    self._nodes_expanded += 1
    is_end, winner = board.is_end(self.player)
    if is_end:
        return INF if winner == self.player else -INF
    if depth >= self.max_depth:
        return self.heuristic(board, self.player)

    value = -INF
    for action in self._ordered_actions(board, self.player, maximize=True):
        child = board.clone()
        child.play(action, self.player)
        value = max(value, self._min_value(child, depth + 1, alpha, beta))
        if value >= beta:
            return value  # poda beta
        alpha = max(alpha, value)
    return value
